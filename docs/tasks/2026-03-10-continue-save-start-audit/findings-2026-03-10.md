# Formal Audit Report: continue/save/start

## 审计对象

1. skill 对象
   - `skills/vibe-continue/SKILL.md`
   - `skills/vibe-save/SKILL.md`
2. workflow 对象
   - `.agent/workflows/vibe-start.md`

## 真源基线

本次审计对照以下真源与实现证据：

1. `docs/standards/skill-standard.md`
2. `docs/standards/command-standard.md`
3. `docs/standards/shell-capability-design.md`
4. `docs/standards/git-workflow-standard.md`
5. `docs/standards/worktree-lifecycle-standard.md`
6. `docs/standards/glossary.md`
7. `bin/vibe`、`lib/flow_help.sh`、`lib/flow_status.sh`、`lib/task_write.sh`
8. `.agent/governance.yaml`
9. `alias/worktree.sh`

## Findings

### Blocking

1. `skills/vibe-save/SKILL.md` 把只读查询命令描述成“自动对齐”能力。
   - 证据：`skills/vibe-save/SKILL.md` 第 26 行写明“物理层状态已通过 `vibe flow status` 自动对齐”。
   - 对照：`docs/standards/command-standard.md` 第 110-128 行要求运行时状态只能查询展示，`vibe` Shell 不是 workflow engine；`docs/standards/shell-capability-design.md` 第 1-8 节禁止把 Shell 描述成隐藏工作流执行器。
   - 实现：`lib/flow_status.sh` 第 31-48 行仅仅读取 dashboard 并打印 open flow 数据，没有任何写入或修复路径。
   - 结论：这是把查询命令改写成隐式修复承诺，属于越权描述，必须移除。

### Missing Reference

1. `skills/vibe-continue/SKILL.md` 缺少与其语义范围匹配的 standards 引用。
   - 证据：文件内未引用任何 `docs/standards/*`，但全文承担了 shared-state、flow、worktree、task、workflow 边界说明。
   - 对照：`docs/standards/skill-standard.md` 第 34-45 行要求 workflow/skill 语义以权威层为准；`vibe-skill-audit` 审查清单要求这类 skill 至少显式对齐 `skill-standard`、`command-standard`、`shell-capability-design`、`git-workflow-standard`、`worktree-lifecycle-standard`。

2. `skills/vibe-save/SKILL.md` 缺少与其语义范围匹配的 standards 引用。
   - 证据：文件内未引用任何 `docs/standards/*`，但全文覆盖 shared-state 写回、缓存分层、flow/task/worktree 边界与治理 hook。
   - 对照：同上；其语义范围已经超出单纯“保存会话摘要”。

3. `.agent/workflows/vibe-start.md` 缺少与其 workflow 语义相匹配的 standards 引用。
   - 证据：文件内未引用任何 `docs/standards/*`，但直接规定 plan/task/worktree/identity/execution 规则。
   - 对照：`docs/standards/skill-standard.md` 第 39-45 行要求 `.agent/workflows/` 作为跨 Agent 语义层时，应服从真源，不应自成第二套定义。

### Capability Gap

1. `skills/vibe-save/SKILL.md` 依赖 `.agent/governance.yaml` 的 `flow_hooks.done` 编排保存行为，但仓库没有对应执行器证据。
   - 证据：`skills/vibe-save/SKILL.md` 第 187-189 行声明“触发 Governance Hook”并称保存行为受 `flow_hooks.done` 编排；第 236 行进一步写成“手动 `/vibe-save` + Hook 提醒”。
   - 对照：`.agent/governance.yaml` 第 31-42 行确实声明了 `done: [vibe-save, vibe-check]`。
   - 反证：仓库实现中未找到读取 `flow_hooks` 并执行这些 hook 的 shell/runtime 路径；`docs/standards/vibe-engine-design.md` 第 9 行明确写出“`flow_hooks` 被记录了，但无人执行”。
   - 结论：应显式声明为 capability gap，而不是继续保留自动编排承诺。

2. `skills/vibe-continue/SKILL.md` 与 `.agent/workflows/vibe-start.md` 都把 `wtinit <agent>` 当成恢复/执行前置，但当前仓库没有可验证的真实入口。
   - 证据：`skills/vibe-continue/SKILL.md` 第 54 行、`.agent/workflows/vibe-start.md` 第 27 行都要求在签名不匹配时先执行 `wtinit <agent>`。
   - 实现核对：`bin/vibe` 帮助面与 `vibe flow -h`、`vibe task -h` 中都没有该入口；全文搜索也未找到 `wtinit()`、`alias wtinit` 或等价实现，只有 `tests/flow/test_flow_bind_done.bats` 第 89 行存在测试 stub。
   - 结论：当前不是“发现性差”，而是缺少可验证实现，应归类为 capability gap。

### Drift Warning

1. `.agent/workflows/vibe-start.md` 把 `.agent/context/task.md` 提升成执行图纸来源，与 save/standard 中对其“本地 handoff 草稿”的定位冲突。
   - 证据：`.agent/workflows/vibe-start.md` 第 15、20-21 行要求从 `plan.md` 或当前 `task.md` 中读取 checklist 并据此步进执行。
   - 对照：`skills/vibe-save/SKILL.md` 第 50、130 行把 `.agent/context/task.md` 定义为当前 worktree 的未跟踪草稿缓存；`docs/standards/git-workflow-standard.md` 第 264 行明确规定它“只作为 skill 之间的短期 handoff 记录，不是共享真源”。
   - 结论：workflow 对 `task.md` 的角色描述需要回退为 handoff/context，而不是正式执行图纸。

2. `skills/vibe-continue/SKILL.md` 的开头话术仍把 `task.md` / `memory/` 写成恢复入口，但正文实际已经切到 pointer/cache/shared-state 优先。
   - 证据：文件描述和引言第 3、8 行仍写“Reads task.md and memory/ to restore state”；但第 20-32 行把 `.vibe/current-task.json`、`.vibe/focus.md`、`.vibe/session.json` 与 shared registry/task memory 列为优先读取对象，第 96-97 行又把 `.agent/context/task.md` 降为缺失时的本地回填。
   - 实现：`lib/task_write.sh` 第 122-131 行确实会生成 `.vibe/current-task.json`、`.vibe/focus.md`、`.vibe/session.json`。
   - 结论：这是文本前后层级不一致，属于 drift，不应继续把旧描述留在 summary 区。

## 修复原则

> 修复目标是让 skill 或 workflow 回归真源，不是让真源去适配 skill 文本。

1. 删除所有把只读 shell 查询包装成“自动对齐”“自动修复”的表述。
2. 为 continue/save/start 补齐与其语义范围匹配的 standards 引用，不在 skill/workflow 内重复定义术语。
3. 对尚无执行器的 hook 与尚无入口的 `wtinit`，统一改写为 capability gap 或待人工触发说明，不再写成既有能力。
4. 明确 `.agent/context/task.md` 只是本地 handoff/草稿缓存；正式执行图纸应来自 plan 类文档或显式任务计划，而不是临时缓存。
