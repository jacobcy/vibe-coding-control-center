# AI Audit Review: continue/save/start 审计清单

> 本文档不是审计结果，而是后续执行正式审计时必须遵守的检查清单与判定基线。

> 本清单已于 2026-03-10 执行，正式 findings 见 [findings-2026-03-10.md](findings-2026-03-10.md)。

> 文中出现的 `.vibe/current-task.json`、`.vibe/focus.md`、`.vibe/session.json` 仅用于记录修复前的审计证据与判定基线，不代表当前运行时实现仍依赖这些本地缓存文件。

## 前提声明

本次审计必须先声明以下前提：

- 我们是在修复 skill 和 workflow 文本中的错误、漂移、越权描述与假承诺。
- 我们不是把 skill 文本当作真源，再反过来要求标准文档或 shell 实现去适配 skill。
- 真源优先级固定为：标准文档 > 实际 shell 能力与帮助面 > 已验证实现 > skill/workflow 文本。

## 审计对象与边界

本轮对象分为两类，必须分开审计：

1. skill 对象
   - `skills/vibe-continue/SKILL.md`
   - `skills/vibe-save/SKILL.md`
2. workflow 对象
   - `.agent/workflows/vibe-start.md`

补充约束：

- `start` 当前不是 `skills/` 下的 skill，而是 workflow 入口。
- 若同时引用 `.claude/commands/*` 或其他 IDE 适配层，必须把它们视为“适配层镜像”，不得作为真源。

## 审计真源基线

正式审计时，至少对照以下真源：

1. `docs/standards/v3/skill-standard.md`
2. `docs/standards/v3/command-standard.md`
3. `docs/standards/v3/python-capability-design.md`
4. `docs/standards/v3/git-workflow-standard.md`
5. `docs/standards/v3/worktree-lifecycle-standard.md`
6. `bin/vibe` 当前帮助面与实际 `lib/*.sh` 实现

必要时补充：

- `docs/standards/glossary.md`
- `docs/standards/action-verbs.md`
- `.agent/governance.yaml`
- `lib/task_write.sh` 中 `.vibe/*` 缓存的真实生成逻辑

## 审计清单

### 1. 对象类型判定

- 先判断目标是 skill、workflow，还是 Claude/IDE 适配层。
- skill 只按 skill 边界审，workflow 只按 workflow 入口语义审。
- 不允许把 workflow 文本的错误算成 shell 命令设计缺陷，除非 shell 帮助面也同样错误。

### 2. 标准引用完整性

- 检查目标文件是否引用了与其语义范围匹配的标准真源。
- continue/save 这类涉及共享状态、flow/worktree 边界、shell 命令的文本，至少应覆盖：
  - `docs/standards/v3/skill-standard.md`
  - `docs/standards/v3/command-standard.md`
  - `docs/standards/v3/python-capability-design.md`
  - `docs/standards/v3/git-workflow-standard.md`
  - `docs/standards/v3/worktree-lifecycle-standard.md`
- 若缺失，结论归类为 `Missing Reference`。

### 3. 命令现实性检查

- 凡是文中点名的 `vibe <domain> <subcommand>`、flag、alias，都必须在帮助面或实现里找到证据。
- 若对象提到 `wtinit`、`vibe task update --next-step`、`vibe flow status`、`vibe flow done` 等入口，必须确认当前仓库内存在可用路径。
- 若能力文本存在，但实际入口不可达或无当前实现，结论归类为 `Capability Gap`。

### 4. 共享状态边界检查

- 检查文本是否鼓励 agent 直接修改 `.git/vibe/*.json`。
- 检查文本是否把 shell 读操作描述成“自动对齐”“自动修复”。
- 检查 skill 是否替 shell 做了隐藏 workflow 或共享状态编排承诺。
- 若存在越权描述，结论归类为 `Blocking`。

### 5. 真源与缓存分层检查

- 明确区分共享真源、本地缓存、本地 handoff 草稿。
- 对 continue/save，重点检查：
  - `.vibe/current-task.json`
  - `.vibe/focus.md`
  - `.vibe/session.json`
  - `.agent/context/task.md`
- 若文本把未跟踪草稿抬升成共享真源，或把共享真源降格成临时缓存，归类为 `Drift Warning` 或 `Blocking`，视严重程度而定。

### 6. 自动化承诺与执行器检查

- 凡是文本声称“会自动处理”“由 hook 编排”“会刷新缓存”，都必须找到代码层执行路径。
- 重点核对 `.agent/governance.yaml` 中的 `flow_hooks.done` 是否真有 shell 或运行时执行器。
- 只有配置、没有执行入口时，统一归类为 `Capability Gap`。

### 7. flow / task / worktree 语义对齐检查

- 检查 continue/save/start 是否混淆 `flow`、`task`、`branch`、`worktree`、`workflow`。
- 对 continue/save，检查它们是否把当前 worktree 恢复、任务绑定、共享 task 更新混为一谈。
- 对 start，检查它是否把 workflow 入口误写成 shell 能力，或对 shell 提出不存在的前置要求。

### 8. 身份与签名修正检查

- 若 continue/start 要求执行 `wtinit <agent>`，必须核对：
  - 该入口在仓库语义里是否真实存在
  - 它是否是合理前置，而不是把环境修正强塞给所有恢复/执行流程
- 若只是约定存在但发现性差、入口层级混乱，可归类为 `Drift Warning`。

### 9. `task.md` 角色自洽性检查

- 检查 save 对 `.agent/context/task.md` 的定义，是否与 start 对其用途的描述一致。
- 若 save 把它定义为当前 worktree 的未跟踪草稿，而 start 又把它当作正式执行 checklist 来源，必须报 `Drift Warning`。

### 10. 结论分类约束

最终发现只能落在四桶中：

- `Blocking`
- `Missing Reference`
- `Capability Gap`
- `Drift Warning`

禁止使用“建议优化”“感觉不一致”这类无判定力措辞。

## 建议的审计输出结构

正式审计报告应按以下顺序输出：

1. 审计对象
2. 真源基线
3. Findings
   - Blocking
   - Missing Reference
   - Capability Gap
   - Drift Warning
4. 修复原则

修复原则必须显式包含以下一句：

> 修复目标是让 skill 或 workflow 回归真源，不是让真源去适配 skill 文本。

  - 证据：`skills/vibe-save/SKILL.md` 第 26 行写明“物理层状态已通过 `vibe flow status` 自动对齐”。
  - 对照：`docs/standards/v3/command-standard.md` 第 110-128 行要求运行时状态只能查询展示，`vibe` Shell 不是 workflow engine；`docs/standards/v3/python-capability-design.md` 第 1-8 节禁止把 Shell 描述成隐藏工作流执行器。
  - 实现：`lib/flow_status.sh` 第 31-48 行仅仅读取 dashboard 并打印 open flow 数据，没有任何写入或修复路径。
  - 结论：这是把查询命令改写成隐式修复承诺，属于越权描述，必须移除。

  - 证据：文件内未引用任何 `docs/standards/*`，但全文承担了 shared-state、flow、worktree、task、workflow 边界说明。

### Capability Gap

1. `skills/vibe-save/SKILL.md` 依赖 `.agent/governance.yaml` 的 `flow_hooks.done` 编排保存行为，但仓库没有对应执行器证据。
  - 证据：`skills/vibe-save/SKILL.md` 第 187-189 行声明“触发 Governance Hook”并称保存行为受 `flow_hooks.done` 编排；第 236 行进一步写成“手动 `/vibe-save` + Hook 提醒”。
  - 对照：`.agent/governance.yaml` 第 31-42 行确实声明了 `done: [vibe-save, vibe-check]`。
  - 反证：仓库实现中未找到读取 `flow_hooks` 并执行这些 hook 的 shell/runtime 路径；`docs/standards/v3/vibe-engine-design.md` 第 9 行明确写出“`flow_hooks` 被记录了，但无人执行”。
  - 结论：应显式声明为 capability gap，而不是继续保留自动编排承诺。

2. `skills/vibe-continue/SKILL.md` 与 `.agent/workflows/vibe-start.md` 都把 `wtinit <agent>` 当成恢复/执行前置，但当前仓库没有可验证的真实入口。
  - 证据：`skills/vibe-continue/SKILL.md` 第 54 行、`.agent/workflows/vibe-start.md` 第 27 行都要求在签名不匹配时先执行 `wtinit <agent>`。
  - 实现核对：`bin/vibe` 帮助面与 `vibe flow -h`、`vibe task -h` 中都没有该入口；全文搜索也未找到 `wtinit()`、`alias wtinit` 或等价实现，只有 `tests/flow/test_flow_bind_done.bats` 第 89 行存在测试 stub。
  - 结论：当前不是“发现性差”，而是缺少可验证实现，应归类为 capability gap。

### Drift Warning

1. `.agent/workflows/vibe-start.md` 把 `.agent/context/task.md` 提升成执行图纸来源，与 save/standard 中对其“本地 handoff 草稿”的定位冲突。
  - 证据：`.agent/workflows/vibe-start.md` 第 15、20-21 行要求从 `plan.md` 或当前 `task.md` 中读取 checklist 并据此步进执行。
  - 对照：`skills/vibe-save/SKILL.md` 第 50、130 行把 `.agent/context/task.md` 定义为当前 worktree 的未跟踪草稿缓存；`docs/standards/v3/git-workflow-standard.md` 第 264 行明确规定它“只作为 skill 之间的短期 handoff 记录，不是共享真源”。
  - 结论：workflow 对 `task.md` 的角色描述需要回退为 handoff/context，而不是正式执行图纸。

2. `skills/vibe-continue/SKILL.md` 的开头话术仍把 `task.md` / `memory/` 写成恢复入口，但正文实际已经切到 pointer/cache/shared-state 优先。
  - 证据：文件描述和引言第 3、8 行仍写“Reads task.md and memory/ to restore state”；但第 20-32 行把 `.vibe/current-task.json`、`.vibe/focus.md`、`.vibe/session.json` 与 shared registry/task memory 列为优先读取对象，第 96-97 行又把 `.agent/context/task.md` 降为缺失时的本地回填。
  - 实现：`lib/task_write.sh` 第 122-131 行确实会生成 `.vibe/current-task.json`、`.vibe/focus.md`、`.vibe/session.json`。
  - 结论：这是文本前后层级不一致，属于 drift，不应继续把旧描述留在 summary 区。

## 修复原则

> 修复目标是让 skill 或 workflow 回归真源，不是让真源去适配 skill 文本。

1. 删除所有把只读 shell 查询包装成“自动对齐”“自动修复”的表述。
2. 为 continue/save/start 补齐与其语义范围匹配的 standards 引用，不在 skill/workflow 内重复定义术语。
3. 对尚无执行器的 hook 与尚无入口的 `wtinit`，统一改写为 capability gap 或待人工触发说明，不再写成既有能力。
4. 明确 `.agent/context/task.md` 只是本地 handoff/草稿缓存；正式执行图纸应来自 plan 类文档或显式任务计划，而不是临时缓存。
