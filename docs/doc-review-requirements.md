# 文档审核需求书（并行 Worktree 工作流）

## 目标
- 统一项目文档标准，确保信息一致、可执行、可维护。
- 支持并行 worktree + PR 工作流：每个 agent 在独立 worktree 提交文档更新，Claude 在 main 负责审核与合并。

## 适用范围
- `README.md`、`DEVELOPER.md`、`CONTRIBUTING.md`、`CHANGELOG.md`
- `docs/` 下所有 Markdown 文档
- `templates/` 下的 PRD/PR/Spec 模板
- `lib/vibe-help.md`（CLI 帮助文档）

## 角色与责任
- **开发 agent（各自 worktree）**：负责文档更新与 PR 提交。
- **Claude（main 分支）**：负责文档审核、测试验证、合并与发布记录更新。

## 审核标准
### 1. 一致性
- 关键术语、命令名、脚本路径必须与仓库一致。
- 不同文档对同一流程的描述不得冲突。
- 模板中的字段名称与实际流程/脚本一致。

### 2. 可执行性
- 命令示例应可直接运行（含前置条件）。
- 提到的文件路径必须存在或明确说明“待创建”。
- 工作流步骤须完整、无跳步。

### 3. 可维护性
- 说明内容避免硬编码（如绝对路径），优先引用 `VIBE_ROOT` / `VIBE_HOME`。
- 文档结构清晰，标题层级合理。
- 变更记录应更新 `CHANGELOG.md`（如果影响用户使用）。

### 4. 风险与边界
- 明确“当前实现”和“目标态”区分。
- 标注实验/未实现功能。
- 重要操作需提示风险（如覆盖配置、删除文件）。

## PR 提交要求（最简）
PR 描述必须包含一行摘要：
`变更摘要 | 测试说明 | 风险点`

## Claude 审核清单（main 分支）
- [ ] 文档一致性检查
- [ ] 示例命令可执行性检查
- [ ] 变更影响范围确认
- [ ] 必要时更新 `CHANGELOG.md`

## 归档
- 每次审核记录保存在 `docs/reviews/` 下，文件名格式：`review-YYYYMMDD-HHMM.md`
