# migrate-to-v3 Design

## Context

Vibe Center 2.0 当前架构：
- **单层模块**：所有功能模块平铺在 lib/ 目录，缺乏清晰的层次划分
- **耦合度**：模块间存在循环依赖和隐式耦合
- **配置散落**：配置逻辑与业务逻辑混合
- **扩展困难**：添加新能力需要修改多个文件

v3 目标架构：Control Plane（控制平面）模式
- **三层架构**：核心层（路由/生命周期）→ 服务层（能力模块）→ 工具层（通用工具）
- **能力注册**：所有能力通过注册中心统一管理
- **配置分离**：配置文件与代码完全分离
- **钩子扩展**：通过生命周期钩子支持扩展

## Goals / Non-Goals

**Goals:**
- 建立清晰的三层架构（核心/服务/工具）
- 实现能力注册中心，统一管理所有能力模块
- 将配置管理从代码中完全分离
- 建立生命周期钩子系统
- 保持 CLI 接口的向后兼容性

**Non-Goals:**
- 不重写所有功能，优先迁移而非重写
- 不改变外部工具依赖（bats, jq, curl, gh）
- 不修改用户可见的命令行界面（保持别名兼容）
- 不引入新的编程语言或框架

## Decisions

### Decision 1: 三层架构设计

**选择**：采用核心层、服务层、工具层的三层架构

**方案对比**：
- **方案 A（选中）**：三层架构
  - 核心层：命令路由、生命周期管理
  - 服务层：能力模块（control-plane-core, capability-registry 等）
  - 工具层：utils.sh 等通用工具
- **方案 B**：微内核 + 插件架构
  - 优点：扩展性更强
  - 缺点：过度设计，当前规模不需要
- **方案 C**：保持现状，仅重构模块
  - 优点：改动最小
  - 缺点：无法建立清晰的架构边界

**理由**：
- 三层架构平衡了清晰性和复杂度
- 符合 Vibe Center 的"拒绝过度工程化"原则
- 为未来扩展留出空间，但不引入不必要的复杂性

### Decision 2: 能力注册中心实现

**选择**：基于文件系统的轻量级注册中心

**方案对比**：
- **方案 A（选中）**：基于文件系统的注册
  - 每个能力在 `lib/capabilities/` 下有独立的 .sh 文件
  - 通过命名约定自动发现能力
  - 使用 source 引入能力模块
- **方案 B**：中心化注册表
  - 维护一个 registry.sh 文件，显式注册所有能力
  - 优点：集中管理
  - 缺点：每次新增能力需要修改注册表
- **方案 C**：动态加载
  - 运行时扫描并加载能力
  - 优点：完全自动化
  - 缺点：性能开销，难以静态分析

**理由**：
- 文件系统方案最符合 Shell 脚本的惯用模式
- 命名约定提供结构化，同时保持灵活性
- 无需额外的注册步骤，新增能力只需创建文件

### Decision 3: 配置管理分离

**选择**：环境变量 + YAML 配置文件

**方案对比**：
- **方案 A（选中）**：环境变量 + YAML
  - 敏感信息用环境变量
  - 非敏感配置用 YAML 文件
  - 使用 yq 解析 YAML（类似 jq）
- **方案 B**：纯 Shell 变量
  - 所有配置放在 .sh 文件中
  - 优点：无依赖
  - 缺点：难以支持复杂配置结构
- **方案 C**：JSON 配置
  - 使用 JSON 作为配置格式
  - 优点：已有 jq 依赖
  - 缺点：JSON 不支持注释，可读性差

**理由**：
- YAML 是最平衡的选择：可读性好、支持复杂结构、注释友好
- yq 是成熟的工具，与 jq 使用体验一致
- 环境变量用于敏感信息符合 12-factor app 原则

### Decision 4: 生命周期钩子实现

**选择**：基于函数命名的钩子系统

**方案对比**：
- **方案 A（选中）**：函数命名约定
  - 定义 hook_{before|after}_{command} 函数
  - 在命令执行前后自动调用
  - 简单直接，无额外依赖
- **方案 B**：事件总线
  - 发布-订阅模式
  - 优点：解耦性更强
  - 缺点：对 Shell 脚本来说过于复杂
- **方案 C**：配置文件驱动
  - 在配置文件中定义钩子
  - 优点：可配置性强
  - 缺点：增加配置复杂度

**理由**：
- 函数命名约定最符合 Shell 脚本的简洁性
- 无需额外的配置或依赖
- 易于理解和维护

## Risks / Trade-offs

### Risk 1: 迁移过程中的功能回归
**风险**：大规模重构可能引入 bug，导致现有功能失效
**缓解**：
- 分阶段迁移：先迁移核心层，再逐个迁移能力模块
- 每个阶段都运行完整的测试套件
- 保持向后兼容的命令别名

### Risk 2: 性能开销
**风险**：三层架构可能引入额外的函数调用开销
**缓解**：
- Shell 脚本的性能瓶颈通常在外部命令，而非函数调用
- 使用 lazy loading 仅在需要时加载能力模块
- 性能测试：对比迁移前后的命令执行时间

### Risk 3: 学习曲线
**风险**：新架构对贡献者来说需要学习成本
**缓解**：
- 完善的文档：STRUCTURE.md 更新架构说明
- 清晰的代码注释：每个层次的职责说明
- 示例：提供能力模块的模板和示例

### Trade-off: 灵活性 vs 简洁性
**权衡**：三层架构比平铺结构更复杂，但提供了更好的扩展性
**决策**：接受适度的复杂性以换取长期的维护性和扩展性

## Migration Plan

### Phase 1: 核心层搭建（Week 1）
1. 创建 `lib/core/` 目录结构
2. 实现命令路由器（router.sh）
3. 实现生命周期管理器（lifecycle.sh）
4. 建立能力注册中心框架（capability_registry.sh）

### Phase 2: 能力模块迁移（Week 2-3）
1. 识别现有功能模块的能力边界
2. 将功能迁移到 `lib/capabilities/` 下的独立模块
3. 每个能力模块实现 `init()` 和 `register_hooks()` 接口
4. 更新测试以覆盖新架构

### Phase 3: 配置分离（Week 3）
1. 创建 `config/environments/` 目录结构
2. 将硬编码配置迁移到 YAML 文件
3. 实现配置加载器（config_loader.sh）
4. 更新文档说明配置方式

### Phase 4: 集成测试与优化（Week 4）
1. 端到端测试：所有 CLI 命令
2. 性能测试：对比迁移前后的执行时间
3. 文档更新：STRUCTURE.md, AGENTS.md, CLAUDE.md
4. 向后兼容性验证

### Rollback Strategy
- **Git Worktree 隔离**：所有开发在独立的 worktree 中进行
- **Feature Branch**：v3 功能在 v3-control-plane 分支开发
- **渐进式合并**：每个 Phase 完成后才合并到 main
- **兼容层**：保留 v2 的命令别名，确保现有用户脚本不受影响

## Open Questions

1. **yq 依赖**：是否需要将 yq 加入必须依赖，还是保留为可选依赖并回退到环境变量？
2. **能力模块粒度**：是否需要支持子能力（sub-capabilities）的概念？
3. **配置热更新**：是否需要支持运行时配置更新，还是只在启动时加载？
4. **钩子优先级**：当多个能力注册同一钩子时，如何确定执行顺序？
