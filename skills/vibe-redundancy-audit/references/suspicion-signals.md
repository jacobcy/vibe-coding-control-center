# Suspicion Signals

本文件列出 `vibe-redundancy-audit` 关注的高价值可疑点类型。

## 1. 迁移后残留

典型信号：

- 新 usecase / helper 已引入
- 对应 command 文件仍保留大量业务判断
- command 文件 LOC 明显下降不够，仍高于团队期望
- 同一职责同时在 command 和 usecase 中出现

人工确认重点：

- 旧路径是否仍承担真实兼容责任
- 是否只是本轮重构尚未完成
- 是否可以在下一轮继续下沉或拆分

## 2. 重复编排

典型信号：

- 多个命令重复构建相同 request / context / error translation
- 相似的 side-effect 顺序在多个模块平行实现
- 同一类参数解析后进入相似逻辑，但未抽出共享能力

人工确认重点：

- 是否属于稳定模式，值得提炼
- 是否只是不同命令的必要差异

## 3. 可疑兼容逻辑

典型信号：

- deprecated alias 仅保留壳层
- fallback 分支长期存在
- 历史命令名称仍保留，但代码侧调用面极薄

人工确认重点：

- 是否仍有外部用户依赖
- 是否已有迁移窗口或废弃计划

## 4. 疑似无人使用的业务逻辑

典型信号：

- 符号调用面接近零
- 不在公开 CLI 入口
- 不在测试主路径
- 不承担框架约定入口

人工确认重点：

- 是否通过反射、配置、CLI、模板或外部约定触发
- 是否只是当前分析工具未覆盖到引用

## 5. 共享能力缺失

典型信号：

- 相似模式反复出现
- 文件职责相似但未共用组件
- 同一种错误翻译、参数解析、状态判定被重复书写

人工确认重点：

- 是否应沉淀为 usecase / service / helper
- 抽取后是否真的能减少认知负担，而不是增加间接层
