# 执行模式说明

本文档描述 vibe-review-pr 的四种执行模式及其决策逻辑。

## 四种执行模式

### ask-each（默认，最安全）

**决策者**: 用户

**workflow**: `review → 询问用户 → 执行用户选择`

**风险**: 无（用户自己决定）

**适用场景**: 所有情况都适用

---

### auto-decide（复杂度自动决策）

**决策者**: team-lead

**workflow**: `review → lead 判断 → fix 或 comment`

**风险**: 中（依赖 lead 的复杂度评估）

**适用场景**: 用户明确授权自动决策

---

### auto-fix（自动修复）

**决策者**: 自动化流程

**workflow**: `review → fix → commit → comment`

**风险**: 高（可能引入新问题）

**适用场景**: 用户明确要求自动修复

**约束**:
- 仅适用于改动行数 < 200 且无安全相关改动的 PR
- 修复前必须备份当前状态（git stash）
- 修复后必须运行测试验证

---

### comment-only（只写 comment）

**决策者**: 无（只记录）

**workflow**: `review → comment`

**风险**: 无

**适用场景**: 用户要求只审核不修复

---

## 复杂度计算

auto-decide 模式通过复杂度评分（0-1）决定执行路径：

| 指标         | 权重 | low (0分)            | medium (0.5分)             | high (1分)                  |
| ------------ | ---- | -------------------- | --------------------------- | --------------------------- |
| 改动行数     | 0.3  | < 50 行              | 50-200 行                   | > 200 行                    |
| 文件数       | 0.2  | 1-2 个文件           | 3-5 个文件                  | > 5 个文件                  |
| 安全相关     | 0.4  | 无安全标签           | 涉及认证/授权               | 安全修复或敏感数据处理      |
| 测试覆盖     | 0.1  | 有完整测试           | 部分测试                    | 无测试                      |

**计算公式**: `复杂度得分 = Σ (指标得分 × 权重)`

---

## 决策矩阵

| 条件                                           | 执行路径                       |
| ---------------------------------------------- | ------------------------------ |
| 复杂度得分 < 0.3 且无安全相关改动              | auto_fix                      |
| 复杂度得分 >= 0.3 或有安全相关改动             | ask_user（升级为 ask-each）   |
| 复杂度得分 >= 0.7 或 lead 置信度 < 0.6         | comment_only                  |

---

## 执行模式选择建议

| PR 类型                           | 推荐模式                  | 理由                                   |
| --------------------------------- | ------------------------- | -------------------------------------- |
| 简单 PR（< 50 行，无安全相关）    | auto-decide 或 auto-fix   | 低风险，自动化效率高                   |
| 标准 PR（50-200 行）              | ask-each                  | 用户参与决策，平衡效率与安全           |
| 安全 PR（涉及认证/授权）          | ask-each                  | 高风险，必须用户确认                   |
| 大型 PR（> 200 行）               | ask-each 或 comment-only  | 复杂度高，避免自动修复引入问题         |
